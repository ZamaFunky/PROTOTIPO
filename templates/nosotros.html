Sistema de Control de Acceso a Edificios — Análisis del Proyecto

ProyectoPublicado: 2025   ·   Autor: Diego   ·   Categoría: Seguridad & Control de Acceso

Resumen ejecutivo

Este proyecto propone un Sistema de Control de Acceso a Edificios que sustituya el uso de llaves físicas por credenciales digitales y biométricas (huella, reconocimiento facial) y/o identificadores RFID. El objetivo es mejorar la seguridad, la trazabilidad del ingreso y reducir los riesgos de duplicación y pérdida de llaves.

1. Contexto del problema

En muchos entornos corporativos aún se usan llaves físicas para autorizar accesos. Esto genera problemas frecuentes:

  • Pérdida frecuente de llaves y pérdida de control.
  • Duplicación no autorizada por terceros.
  • Falta de trazabilidad: no hay registro fiable de quién entra y cuándo.

2. Objetivo del proyecto

Desarrollar e implementar un sistema fiable y seguro que use credenciales biométricas y/o digitales (tarjetas/RFID, ID, apps móviles) para autorizar el acceso a un edificio, registrar eventos y facilitar la administración centralizada.

3. Análisis según IEEE 830

3.1 Resumen

Se define un sistema que reemplaza llaves físicas por credenciales digitales y biométricas, con módulos de autenticación, registro (log), administración de usuarios y una interfaz web para administración.

3.2 Introducción

El presente documento detalla los requerimientos funcionales y no funcionales, el alcance del sistema y el entorno operativo. Está orientado a responsables de seguridad, administradores TI y desarrolladores que implementarán la solución.

3.3 Antecedentes del cliente

Empresa con múltiples empleados y visitantes diarios, necesita: control de acceso robusto, registro de eventos y opciones para revocar credenciales rápidamente.

3.4 Entorno de operación

  • Hardware: sensores biométricos (lector de huella, cámara para reconocimiento facial), lectores RFID, puertas/cerraduras electrónicas, controladores de puerta (PLC o controlador embebido), servidores locales o en la nube.
  • Software: aplicación web responsiva para administrador y usuarios, API de autenticación, base de datos segura (ej. PostgreSQL), integrations con Active Directory si aplica.

3.5 Justificación social

Mejora de la seguridad laboral, reducción de intrusiones, mayor trazabilidad y adopción de tecnologías eficientes que apoyan ODS 9 y ODS 11.

3.6 Requerimientos normativos y ODS

  • Cumplimiento con normativas de protección de datos (GDPR o leyes locales de datos personales).
  • Alineación con ODS 9 (Industria, innovación) y ODS 11 (Ciudades sostenibles).

4. Arquitectura propuesta

Arquitectura modular compuesta por los siguientes componentes:

  • Frontend: Interfaz web para administradores y panel de acceso para guardias/usuarios. Responsive y accesible.
  • Backend: API REST/GraphQL encargada de autenticación, autorización (roles) y registro de eventos.
  • Base de datos: almacenamiento seguro de usuarios, credenciales, logs de acceso y configuración de dispositivos.
  • Integración hardware: controladores de puerta con conexión segura (TLS/VPN) y drivers para lectores biométricos/RFID.

5. Alcance

  • Control de entrada/salida mediante credenciales.
  • Registro y almacenamiento de eventos (fecha/hora, usuario, método de autenticación, puerta).
  • Gestión de usuarios y credenciales (alta, baja, bloqueo, renovación).
  • Alertas en caso de intentos fallidos, manipulación o hardware desconectado.

6. Técnicas de recolección de datos

Para definir requerimientos se propone:

  • Entrevistas con personal de seguridad y administración.
  • Observación del flujo de acceso en horas pico.
  • Encuestas a empleados (privacidad, preferencia de métodos de acceso).

7. Requerimientos funcionales (ejemplos)

  1. RF-01 Validación de identidad: El sistema debe autenticar usuarios mediante huella, rostro, ID o RFID.
  2. RF-02 Registro de accesos: Registrar fecha, hora, método de autenticación, resultado (permitido/denegado) y puerta.
  3. RF-03 Gestión de credenciales: Alta, baja, bloqueo y renovación de credenciales por administrador.
  4. RF-04 Panel de auditoría: Consultas y exportación de logs (CSV/PDF) por rango de fechas y filtros.

8. Requerimientos no funcionales

  • RNF-01 Seguridad de datos: Encriptación de credenciales y comunicación TLS entre componentes.
  • RNF-02 Alta disponibilidad: SLA objetivo 99.5% para servicios de autenticación.
  • RNF-03 Rendimiento: Autenticación en < 500ms en condiciones normales.
  • RNF-04 Privacidad: Almacenamiento seguro de plantillas biométricas (no imágenes sin consentimiento) y cumplimiento normativo.
  • RNF-05 Usabilidad: Interfaz clara para administradores sin necesidad de conocimientos técnicos avanzados.

9. Lista de cotejo (checklist)

  • ¿Se identificaron todos los actores (usuario, administrador, guardia, visitante)?
  • ¿Se definieron claramente los requerimientos funcionales y no funcionales?
  • ¿Se consideraron aspectos legales y de privacidad?
  • ¿Existe plan de respaldo y recuperación ante fallo?
  • ¿Se planeó la integración física con cerraduras y sensores?

10. Diseño (ALTO NIVEL)

El diseño se plantea en capas:

  1. Presentación: Web app y panel móvil para administración.
  2. Servicios: API de autenticación, servicio de logs, servicio de notificaciones (email/SMS).
  3. Persistencia: Base de datos relacional (PostgreSQL) + almacenamiento de blobs cifrados si aplica.
  4. Integración de hardware: Gateways que conversan con el backend por API segura.

Flujos críticos (ejemplo)

Ingreso con biometría: Lector captura huella → Gateway procesa plantilla → API backend valida → Si OK, orden de apertura a cerradura y registro del evento.

Ingreso con tarjeta RFID: Lector envía ID → Gateway consulta API → Si autorizado, abre puerta y registra evento.

11. Riesgos y mitigaciones

  • Riesgo: Robo o suplantación de credenciales. Mitigación: Multifactores (biometría + tarjeta) y monitoreo de intentos fallidos.
  • Riesgo: Falla de hardware. Mitigación: Redundancia y alertas de salud del dispositivo.
  • Riesgo: Incumplimiento normativo. Mitigación: Asesoría legal, anonimización y consentimiento informado.

12. Plan de despliegue y mantenimiento

Fases sugeridas:

  1. Prueba de concepto en un acceso secundario.
  2. Despliegue piloto en una planta con monitorización 24/7.
  3. Evaluación y ajustes según métricas (falsos positivos, tiempos, feedback).
  4. Despliegue completo y transferencia de conocimiento al equipo de Seguridad TI.

13. Recomendaciones finales

  • Empezar por un piloto pequeño para validar integración con hardware.
  • Registrar y auditar todos los accesos desde día 1.
  • Priorizar la protección de plantillas biométricas y cumplimiento de privacidad.
  • Implementar un plan de recuperación ante incidentes y backups cifrados.
